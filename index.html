1. Спецификация маркеров безопасности RAG

Системные метки и ограничения, применяемые на схеме доступа

🔒 Маскирование PII

Применение: Блок «ЗАКАЗЫ» (БД_2)

Ни один реальный контакт клиента (ФИО, Email, Телефон) не передается в оперативную память LLM. Алгоритмы поиска по шаблонам на уровне API находят конфиденциальные данные и заменяют их на заглушки общего вида [Name], [Email], [Phone].

🔑 Доступ только по ID

Применение: Блок «БРАК» (БД_2)

Предотвращает массовый парсинг отчетов о браке. Менеджер по качеству не может запросить агрегированные данные. Поиск и извлечение файла [id].json строго привязаны к явному верифицированному ID транзакции/пользователя в текущем запросе.

📝 Неудаляемый Лог Изменений

Применение: Блоки БД_3 («Сотрудники» и «Клиенты»)

Защита от внутренних угроз. Любое изменение записи (INSERT/UPDATE/DELETE) Администратором автоматически фиксируется в изолированной СУБД по принципу Append-Only (разрешена только вставка новых строк без возможности стирания истории).

🔑 Подтверждение доступа (MFA / OTP)

Применение: Переход к БД_2 и БД_3

Для получения доступа к заказам или спискам клиентов требуется дополнительный фактор верификации (ввод одноразового OTP-кода, нажатие аппаратного токена или ручное подтверждение согласия с регламентом обработки персональных данных).

2. Описание правил безопасности А и Б (Смысловая концепция)

Алгоритмическая логика обработки информации в RAG Base assist

Правило А: Алгоритм верификации доступа (Access Handshake)

Перед каждым обращением языковой модели к хранилищу RAG, система проводит автоматический невмешательный аудит сессии пользователя. Проверяется ролевое соответствие (User Role Context). При запросе конфиденциальных слоев (БД_2 и БД_3) система запрашивает явное подтверждение прав (OTP-код/MFA). При несовпадении прав сессия мгновенно изолируется с выдачей стандартной формулировки отказа.

Правило Б: Принцип маскирования конфиденциальных данных (PII)

Исключает риск утечки приватной информации за рамки периметра компании через запросы к внешним API-моделям. Все данные, запрашиваемые из базы заказов перед помещением в контекст LLM, прогоняются через автоматический фильтр. Реальные текстовые совпадения с именами, телефонами или адресами перекодируются в безопасные системные маркеры. Ассистент работает только со «стерильными» транзакциями.

3. Цели системных промптов (System Prompts Goals)

Критически важные задачи для настройки поведения LLM в зависимости от роли

🎯 Цель промпта для БД_1 (Пользователь)

  • Жесткое удержание фокуса исключительно на базе знаний общего доступа (faq, условия оплаты/доставки).
  • Абсолютный запрет на раскрытие служебной информации, списков персонала и структуры внутренних систем.
  • Использование неизменяемого вежливого отказа при попытках социальной инженерии (например, на вопрос "Как зовут сотрудника?").

🎯 Цель промпта для Менеджера (БД_2 — ЗАКАЗЫ)

  • Обеспечение корректной аналитической работы с замаскированным PII-контекстом.
  • Предотвращение попыток «раскрытия» или «угадывания» реальных контактов клиентов по косвенным признакам.
  • Изоляция от несанкционированного чтения отчетов о браке (defects) и системных настроек.

🎯 Цель промпта для Менеджера по качеству (БД_2 — БРАК)

  • Локализация поиска: разрешение обрабатывать только тот файл брака, ID которого совпадает с текущей жалобой.
  • Запрет сквозного листинга директории и агрегированного поиска по всем отчетам.
  • Запрет разглашения внутренней структуры логов ИИ клиентам (выдача профессиональных резюме вместо JSON).

🎯 Цель промпта для Администратора (БД_3 + Изменения)

  • Предоставление максимальной свободы сопоставления данных между БД_2 и локальными таблицами БД_3.
  • Обязательное принудительное завершение любого успешного действия по изменению генерацией системной директивы лога.
  • Контроль синтаксической целостности локальных списков сотрудников и клиентов при перезаписи.

Индикаторы и Спецификации безопасности Rag Base assist

🔒 Маскирование PII:

Все персональные данные (ФИО, Email, Телефон) в БД_2 автоматически заменяются на плейсхолдеры [Name] перед передачей контекста в LLM.

🔑 Ограничение по ID / MFA:

Доступ к браку открывается только под конкретный подтвержденный ID пользователя. Менеджеры запрашивают верификацию доступа.

📝 Авто-логирование:

Любое внесение изменений Администратором в БД_3 (Сотрудники/Клиенты) записывается в неудаляемый Append-Only лог-файл аудита.

👁️‍🗨️ Контекстный фильтр:

Используйте интерактивную верхнюю панель симуляции для быстрого аудита доступности баз под каждую роль.
